The initial motivation to generate a sign description was as a QA tool, where a text description could highlight details about the symbols that have semantic relevance but might not be visually distinguished, for instance by indicating that a right-hand or left-hand symbol has been used. A semantic text description might also provide meaningful non-visual access to SignWriting, potentially supporting deaf-blind people who use sign language. Text descriptions may also help people learning SignWriting, scaffolding the process of learning the full symbol set.

A sign description is an ordered set of symbol descriptions. The order is, I think, important for understanding the sign description. Therefore the order is defined by the temporal sequence of the sign.

I created the ability to generate a temporal sequence because not all signs in the wild have it, but it is needed for sorting and for the sign description algorithm. A preferred temporal sequence is already described which this mostly follows, but (to see if it proves useful) the default order places movement symbols immediately after the corresponding hand symbol, based on left- and right-handedness.

Algorithmic generation of the temporal sequence might be considered less ideal than manual creation, so it is intended for when the sequence is needed and hasn't been provided. But I think the algorithm mostly produces desired results, and could be used to get users started on adding the sequence to the sign.

From what I have seen, sign dictionaries mostly provide right-hand dominant signs. If users want a left-hand dominant version they must recreate the sign. This function changes signs from right- to left-hand dominant and vice versa by swapping the position of symbols across the centre of the sign, and mirroring symbols and changing symbol handedness.

Symbol descriptions are the heart of sign descriptions. The approach starts with the name of the base symbol and determines additional information about the symbol's plane, handedness, orientation, and movement direction (as applicable). Written labels of these characteristics are joined with punctuation into a complete description of the symbol.

Currently, the "handedness" of the symbol is before the base symbol name, and other aspects of the description are after the base symbol name. Although this is not implemented yet, I think this function should also have the ability to customize the order of description components, and the punctuation used to clarify grouping. This ability might be important to make symbol descriptions as natural as possible in various human languages.

If symbols apply to the right hand or the left hand, this function allows that to be switched. This supports flipping a sign's dominant hand. It could also be an editing feature, for use when users select a symbol of the wrong handedness and want to correct it.

In the base package, many symbols can have their left-right orientation mirrored. When working on mirroring hand dominance of signs, I found additional symbols that seemed they could be mirrored, but with a different mirroring algorithm. This function provides mirror algorithms for a number of additional symbol ranges.
